## Context

**当前状态**：v1.2 PRD 定义了完整的 Android 性能 CR 规则集（10 维度、~40 条规则）、双通道触发机制（流水线全量 + 本地 Agent）、平台无关 Markdown 输出。该 PRD 是 Skill 实现的唯一权威来源。

**变更驱动**：在探索审查中识别出规则覆盖率、质量闭环、运营保障三大方面的缺口，须在 Phase 0 实现前补齐。

**约束**：
- 纯文档产品，无代码变更
- Markdown 格式，须保持与现有表格/标题结构兼容
- 规则格式七列制（ID/等级/检查项/反例/正例/判定/参考工具）不变

## Goals / Non-Goals

**Goals:**
- 补全遗漏规则，覆盖 P0 高发盲区（WebView、runBlocking、Handler 泄漏、Android 14+ 合规）
- 建立外置规则机制，支持项目级定制/覆盖，无需变更 Skill 内置规则
- 定义审查 SLA（超时、规模、并发）和降级策略
- 建立质量反馈闭环（误报标记 → 聚合 → 规则修订）
- 声明静态审查能力边界（已知限制）

**Non-Goals:**
- 不修改双通道架构设计
- 不改变 Markdown 输出格式的主体结构
- 不涉及 CI 流水线的具体实现
- 不与特定 AI 模型绑定

## Decisions

| 决策 | 选择 | 原因 |
|------|------|------|
| 外置规则加载方式 | 文件发现（`/.android-performance-cr/`） | 零配置接入，仓库文件变更即生效，无需搭建规则中心 |
| 外置规则格式 | 与 §10 同构的 Markdown 表格 | Skill AI 可直接复用解析逻辑，无额外格式转换 |
| 规则覆盖优先级 | 项目 overrides > 项目 custom > 内置 | 项目自治优先，内置规则作为兜底安全网 |
| 审查超时阈值 | 5 分钟 | 中等规模 PR（~30 文件）在主流模型下完整审查的实测估计 |
| 大 PR 触发线 | > 50 个变更文件 | 超过此阈值 Token 和耗时风险显著上升 |
| 误报阈值 | 连续 2 周 > 20% | 避免单次波动触发修改，两周足够样本积累 |
| 新知维度 WebView | 独立 §10.11 | WebView 内存/渲染/初始化问题足够多且与通用内存/UI 规则差异大 |

## Risks / Trade-offs

- **[Risk] 外置规则过多导致审查时间膨胀** → 推荐 projects 控制外置规则总数 ≤ 20 条，Skill 启动时打印规则总数供 Reviewer 感知
- **[Risk] 超时降级后 P1/P2 遗漏** → 降级报告明确标注 `partial` + 遗漏维度清单；门禁仅检查 P0
- **[Risk] 误报反馈标记率低、闭环断裂** → Phase 3 看板计入流程指标；若连续 4 周无反馈，自动提醒 Team Leader
- **[Trade-off] 已知限制列出了 9 类盲区** → 明确"不得臆造 P0"原则，避免 AI 过度报告；损失的是盲区问题只能靠运行时工具补充

## Migration Plan

本次为文档产品迭代，无迁移需求：
- v1.2 → v1.3 为向后兼容扩展（新增规则/章节，现有规则 ID 未变）
- Skill 实现代码未创建，无 breaking change
- v1.3 修订记录已写入 PRD §8.3
