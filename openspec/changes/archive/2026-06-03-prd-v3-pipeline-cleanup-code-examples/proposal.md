## Why

PRD v2.0 存在两个结构性问题阻塞 Skill 实际落地：

1. **流水线描述散落全文**：§1-9 中有 12+ 处提及「通道 A/流水线/CI 门禁」，严重稀释了 Skill 的核心定位——**一个纯本地的 AI Code Review 技能，输出标准 Markdown 审查结论文档**。团队反馈流水线细节让 PRD 阅读负担重，且 Phase 2-4 的 CI 集成在现阶段根本不是关注点。

2. **规则反例/正例代码不足**：81 条规则中约 76 条缺少**可编译的 Kotlin 反例+正例代码片段**。AI 模型做 Code Review 时高度依赖模式匹配，概念描述远不如具体代码片段有效。缺少真实代码案例导致：① AI 模型审查准确率低 ② Reviewer 无法快速理解规则的判定边界 ③ 新团队成员学习曲线陡峭。

## What Changes

- **激进清理流水线内容**：§1.2/§1.4/§3.1/§3.2/§3.3/§3.4/§3.6/§5/§6/§7.1/§7.2/§9 中所有流水线/通道 A/CI 门禁描述全部移除或重写为纯本地 Agent 视角。Skill 核心定位改为「开发者本地 AI Code Review 工具」
- **§13 新建「后期展望」章**：将清理出的流水线、CI 门禁、自动化路线图、看板聚合等内容集中到独立展望章节
- **81 条规则全部补全可编译 Kotlin 反例+正例代码**：每条规则至少 2 行代码（反例 ≥ 2 行 + 正例 ≥ 2 行），来自联网搜索的真实生产案例
- **新增代码案例质量标准**：反例必须是可导致明确性能问题的真实代码模式，正例必须是业界推荐的最佳实践写法
- **文档结构重排**：§11 流水线 → §13 后期展望；§7.2 自动化路线图 → 移入 §13；§3.1 双通道 → 简化为单通道（本地 Agent）

## Capabilities

### New Capabilities
- `code-example-standards`: 定义规则反例/正例代码片段的格式、来源和质量标准。每条规则的代码片段 SHALL 来自联网搜索的真实生产案例

### Modified Capabilities
- `rule-checklist`: 81 条规则的反例/正例列全部增强为可编译 Kotlin 代码（至少 2 行反例 + 2 行正例）
- `tool-reference`: 补充工具速查卡中 Compose Compiler Metrics 工具的安装和输出解读
- `output-format`: **BREAKING** 清理输出模板中流水线/通道 A 相关字段，触发通道仅保留 `local-agent`

## Impact

- **影响文件**：`Android 性能 CR AI Code Review 技能 prd文档.md`（全文重构）
- **影响章节**：§1/§3/§5/§6/§7/§11（重写），§10（全部 81 条规则反例/正例增强），新建 §13
- **受影响 Specs**：`rule-checklist`（全部规则代码增强）、`output-format`（字段精简）、`tool-reference`（新增 Compose Compiler Metrics）
- **向后兼容**：所有规则 ID 和等级不变（仅增强描述列）；输出格式字段名保持，但触发通道取值变化
