## Context

PRD v1.3 已建立了 11 个维度的性能审查规则体系（~64 条规则）和完整的 AI Code Review 流程。但在实际落地前的探索中，发现以下结构性问题：

1. **Compose 规则几乎空白**：仅有 UI-05（重组范围）和 UI-07（remember key）两条规则，与现代 Android 应用中 Compose 的普及度严重不匹配
2. **§11 兜底规则长期搁置**：6 条主题已经识别但未落地为可执行规则，其中 16KB 页对齐已是 Google Play 强制合规项
3. **架构维度缺失**：多模块项目常见的跨模块通信开销、循环依赖等性能问题无法审查
4. **工具链信息不足**：§7.1 和 §10 参考工具列仅列名称，AI Skill 无法生成可操作的工具使用建议
5. **流水线内容过早**：Phase 2-4 的 CI/CD 集成将在后期处理，当前 PRD 中混入过多流水线内容

## Goals / Non-Goals

**Goals:**
- §11 全部 6 条主题转正为正式规则，每条补全七列定义
- Compose 性能审查从 2 条扩展到 8 条，形成独立子维度（10.3.A）
- 新建 §10.12 架构与模块化维度（4 条规则）
- 新增 §7.4 工具使用速查卡（9 个工具），支持 AI Skill 在盲区引用
- 将 §7.2 Phase 2-4 + §7.3 流水线内容移至 §11「后期处理：流水线与 CI」
- 补充真实错误案例代码（来自联网搜索的生产环境案例）

**Non-Goals:**
- 不修改现有规则 ID 或等级（仅追加新规则）
- 不实现 Lint/Detekt 规则代码
- 不编写 CI 流水线脚本
- 不修改 §3 双通道架构和输出格式

## Decisions

### D1: Compose 规则放在 §10.3 子维度而非独立 §10.12

| 选项 | 决策 |
|------|------|
| A) 独立 §10.12 Compose 性能 | ```
| B) §10.3.A Compose 专项子维度 | ✅ 选择 |

**理由**：
- Compose 本质属于 UI 渲染维度，归入 §10.3 语义更自然
- 保持维度数量可控（避免超过 12 个主维度）
- AI 审查时按 §6 路径启发式映射到「UI」维度更直观
- 规则 ID 使用 `CP-` 前缀与 `UI-` 前缀区分，方便统计和过滤

### D2: 规则 ID 前缀命名策略

| 维度 | 前缀 | 新增规则 |
|------|------|---------|
| 启动 | `ST-` | ST-07 |
| UI 渲染 | `UI-` | UI-09 |
| Compose 专项 | `CP-` | CP-01~08 |
| 内存 | `MEM-` | MEM-08 |
| 包体 | `PKG-` | PKG-06 |
| 架构模块化 | `ARCH-` | ARCH-01~04 |

**理由**：Compose 使用独立前缀 `CP-`（Compose Performance），避免与 View 体系 `UI-` 规则混淆，便于分体系统计和针对性审查。

### D3: PKG-06 (16KB 页对齐) 定为 P1

| 选项 | 决策 |
|------|------|
| P0（阻塞合入） | |
| P1（应修复） | ✅ 选择 |

**理由**：
- 虽然 Google Play 强制，但国内应用商店尚未要求
- 仅影响使用 NDK/Native `.so` 的项目，纯 Java/Kotlin 项目不触发
- P1 给予项目缓冲期，可在 Issue 中约定修复时间
- 若项目定向 Google Play 且包含 Native 库，团队可通过外置规则 override 升为 P0

### D4: 工具速查卡放 §7.4 而非附录

**理由**：
- §7 已经是「工具与自动化路线图」章节，语义连贯
- 附录 §8 内容已较多（术语、业务指标、修订记录、已知限制）
- AI Skill 在引用工具时通过「参见 §7.4」更简洁

### D5: 流水线内容移至 §11「后期处理」

**理由**：
- §11 原为「后续可选规则」，内容转正后位置空出
- 将 §7.2 Phase 2-4 路线图 + §7.3 流水线集成 + §3.6/3.7 中流水线相关描述整合到 §11
- 保持当前 PRD 聚焦在「Skill AI Code Review 审查规则与流程」
- 后期可直接基于 §11 启动新的 OpenSpec 变更

### D6: 真实错误案例的来源与使用策略

从联网搜索（developer.android.com、掘金、CSDN、proandroiddev.com、readmedium.com 等）搜集了以下典型案例：

| 规则 | 案例来源 | 案例内容 |
|------|---------|---------|
| TH-05 | Death by a Thousand Coroutines (production outage) | `runBlocking { performTask() }` 在 hot path 导致 ANR |
| MEM-02 | 同上 + 官方文档 | `GlobalScope.launch` 在 Activity 销毁后持有引用导致泄漏 |
| MEM-06 | 腾讯云/知乎 Handler 内存泄漏分析 | 非静态内部类 Handler 隐式持有 Activity |
| WV-03 | 掘金 WebView 内存释放全解析 | WebView 未 destroy() 残留数十至数百 MB |
| UI-01 | Android 官方文档 RecyclerView | `notifyDataSetChanged()` vs `DiffUtil` + `ListAdapter` |

案例以反例/正例代码片段的形式写入对应规则的表格中，不单独成章。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| Compose 8 条规则对 AI 模型有学习门槛（需理解 Compose 编译器工作原理） | 每条规则附带清晰的反例/正例 Kotlin 代码；AI 按模式匹配而非深度理解 |
| `ARCH-` 规则偏架构层面，静态 diff 中难以触发 | 仅在改动涉及多模块路径/依赖变更时启用（_scarcity_ 策略） |
| 新增 19 条规则增加审查耗时 | 按 §6 路径启发式筛选相关维度，非全量每 PR 都查 |
| 工具速查卡信息可能过时（工具版本更新） | 仅覆盖稳定 API，不引用具体版本号；建议每半年 review 一次 |
| 流水线后移可能导致读者困惑 §7 的 Phase 标注 | 在 §7 新增明确的「后期处理」引用链接指向 §11 |
